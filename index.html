【设计模式】后知后觉的面向对象编程（二）

Posted on 2019-12-23

【设计模式】后知后觉的面向对象编程（二）

面向对象编程语言的四大特性：封装、抽象、继承、多态
文章部分内容摘录自《设计模式之美》

封装（Encapsulation）

封装也叫作信息隐藏或者数据访问保护。类通过暴露有限的访问接口，授权外部仅能通过类提供的方式（或者叫函数）来访问内部信息或者数据。
对于封装这个特性，我们需要编程语言本身提供一定的语法机制来支持。这个语法机制就是访问权限控制。例子中的 private、public 等关键字就是 Java 语言中的访问权限控制语法。private 关键字修饰的属性只能类本身访问，可以保护其不被类之外的代码直接访问。

封装的意义是什么？它能解决什么编程问题？

如果我们对类中属性的访问不做限制，那任何代码都可以访问、修改类中的属性，虽然这样看起来更加灵活，但从另一方面来说，过度灵活也意味着不可控，属性可以随意被以各种奇葩的方式修改，而且修改逻辑可能散落在代码中的各个角落，势必影响代码的可读性、可维护性。

除此之外，类仅仅通过有限的方法暴露必要的操作，也能提高类的易用性。如果我们把类属性都暴露给类的调用者，调用者想要正确地操作这些属性，就势必要对业务细节有足够的了解。而这对于调用者来说也是一种负担。相反，如果我们将属性封装起来，暴露少许的几个必要的方法给调用者使用，调用者就不需要了解太多背后的业务细节，用错的概率就减少很多。

抽象（Abstraction）

封装主要负责的是如何隐藏信息、保护数据，而抽象负责的是如何隐藏方法的具体实现，让调用者只需要关心方法提供了哪些功能，并不需要知道这些功能是如何实现的。

抽象的意义是什么？它能解决什么编程问题？

如果上升一个思考层面的话，抽象及其前面讲到的封装都是人类处理复杂性的有效手段。在面对复杂系统的时候，人脑能承受的信息复杂程度是有限的，所以我们必须忽略掉一些非关键性的实现细节。而抽象作为一种只关注功能点不关注实现的设计思路，正好帮我们的大脑过滤掉许多非必要的信息。

继承（Inheritance）

继承是用来表示类之间的 is-a 关系，比如猫是一种哺乳动物。从继承关系上来讲，继承可以分为两种模式，单继承和多继承。单继承表示一个子类只继承一个父类，多继承表示一个子类可以继承多个父类，比如猫既是哺乳动物，又是爬行动物。

继承存在的意义是什么？它能解决什么编程问题？

继承最大的一个好处就是代码复用。假如两个类有一些相同的属性和方法，我们就可以将这些相同的部分，抽取到父类中，让两个子类继承父类。这样，两个子类就可以重用父类中的代码，避免代码重复写多遍。不过，这一点也并不是继承所独有的，我们也可以通过其他方式来解决这个代码复用的问题，比如利用组合关系而不是继承关系。

不过，过度使用继承，继承层次过深过复杂，就会导致代码可读性、可维护性变差。为了了解一个类的功能，我们不仅需要查看这个类的代码，还需要按照继承关系一层一层地往上查看“父类、父类的父类……”的代码。还有，子类和父类高度耦合，修改父类的代码，会直接影响到子类。

多态（Polymorphism）

多态是指，子类可以替换父类，在实际的代码运行过程中，调用子类的方法实现。

接口类多态性

定义一个共用的接口类，当传递不同的实现类时，支持动态调用不同的接口；

duck-typing多态性

Duck-typing 实现多态的方式非常灵活。Logger 和 DB 两个类没有任何关系，既不是继承关系，也不是接口和实现的关系，但是只要它们都有定义了 record() 方法，就可以被传递到 test() 方法中，在实际运行的时候，执行对应的 record() 方法。
只要两个类具有相同的方法，就可以实现多态，并不要求两个类之间有任何关系，这就是所谓的 duck-typing，是一些动态语言所特有的语法机制。

多态特性存在的意义是什么？它能解决什么编程问题？

多态特性能提高代码的可扩展性和复用性。如果我们不使用多态特性，我们就无法将不同的集合类型（Array、LinkedList）传递给相同的函数（print(Iterator iterator) 函数）。

【设计模式】后知后觉的面向对象编程（一）

Posted on 2019-11-18

【设计模式】后知后觉的面向对象编程（一）

作为一个从JavaScript入门编程世界的人来说，所谓“面向对象”这个概念，就仿佛从来没真的存在过，它就像那传说中的大侠，江湖处处都有它的传说，却始终没曾有幸得识真面目。
然鹅，
路走多了就会想跑，摔倒了就会想怎么跳。

在自己的工作不再是简单的页面，而逐步进入越来越复杂的模块设计，尤其是Hybrid开发之后，误打误撞的，某些轮廓慢慢的浮现出来，究竟什么是类，什么是面向“对象”，在经历了一轮iOS（Objective-C）和Android（Java）的洗礼下，也逐渐开始明晰起来。
那种定义一个模型，并为其从简单到复杂，私有到公有，继承到多态的感觉，真好啊。

它似乎不再是机械式的通过一系列指令在与机器进行交谈，而是变得有轮廓，有毛发，有自己的呼吸的自生命体（只取决于设计人能力的强大与否）。

扯了这么多玄学，下面还是进入自我总结吧。

什么是面向对象编程

  1. 以类或对象作为组织代码的基本单元
  2. 将封装、抽象、继承、多态四个特性作为代码设计和实现的基石

什么是面向对象分析和面向对象设计

面向对象开发需要经历的三个阶段：面向对象分析（OOA）、面向对象设计（OOD）、面向对象编程（OOP）

  1. 始终围绕着类或对象来做需求设计、分析；
  2. 面向对象分析需要搞清楚要做什么；
  3. 面向对象设计来搞清楚怎么做；
  4. 而面向对象编程则是将需求和设计翻译成为代码的过程；

【设计模式】谈代码质量

Posted on 2019-11-10

【设计模式】谈代码质量

在过去的一两年开发中，也算经历了比较低阶的各种烂代码：
- 复制、粘贴每页抄一遍的
- 变量名字意味不明，随心所欲的
- 过度封装、相互引用到最后妈不认的
- 从来不考虑安全边界的
那些为烂代码痛苦挣扎的日子也就按下不表了，站在当前的界点，能做些什么才能改变这些现状呢，诚然对一个项目来说，工程师的技术能力的重要性毋庸置疑，但往往也会事与愿违，“好刀却不一定有出鞘的机会”，所以除却工程师个人能力的部分，更为科学的代码管理、无关于人的制度层面，也是无可忽视的一部分。

代码的高质量的标准

1. 可读性(readability)

这里我想将可读性放在第一位。一段无法阅读的代码，即便写的再优秀，随着时间的新陈代谢，也只会腐烂在那里，无人问津。
如果一段代码无法被阅读者很好的理解，那么接下来的一切维护、优化、扩展都如无源之水，随时干枯、崩坏，进而制造出无数的新的bug，最终废弃。

如何评价可读性呢，我的标准：

  1. 是否有良好的编码规范（JS中可以使用ESlint进行规范）
  2. 命名是否具有良好的语意性
  3. 注释是否详细（是为读者写的，而不只是为自己应付差事）
  4. 是否简洁易懂（不过度封装、不炫技、实用主义至上）

2. 可维护性(maintainability)

可维护性其实是一个比较综合的概念，代码的可读性好、简洁、注释完备，维护人员很容易就能定位到问题或扩展新功能，就意味着可维护性好。

3. 可扩展性 (extensibility)

在新手程序员经历了无数次对自己那套烂代码重构之后，他们终于开始意识到什么叫扩展性，不再是为每一样需求都定制一个完全新的轮子和零件，而是开始逐渐去建立一些非常基础的模块组件，然后预留未来可预见范围内会加入功能的插槽，已应对后续大部分建立在原代码基础之上的需求。

4. 简洁性(simplicity)

Simple and clean，是我的原则。
很多程序员似乎往往在带着一种蜜汁优越感在进行coding这份工作，仿佛耍着漂亮的花式，全场观众的眼球就会全部停留在自己身上，最后写出那种过半个月自己都看不懂的东西。
对不起，真的很好笑。

简单有效的解决问题，让后来人一眼就能明白你的思路，让一切复杂的问题都能分解成无数段最简单的语句，才是真的功力见深。

5. 可复用性(reusability)、可测试性(testability)

相较于上面4个标准来说，我认为这是一个优秀程序员进阶需要开始关注的问题。
复用性好意味着代码更简洁，可维护性也大幅增加，尤其是在日常的增删改查业务中，如果不注意复用性，往往后果就是大量的copy，paste代码。

而可测试性，就前端来说，还是蛮难的一点，毕竟前端的测试成本空前的高。
但经过自己在leetcode一段时间的练习后，其实深刻感受到了测试的重要性：
- 边界检查（能够大幅提升代码的稳健性）
- 回归检查（能够在每日日常的更新中，保证代码的运行不会出现异常问题）

总结

在应对日常的工作中，把提高代码的可阅读性作为一切一切的基础，是自己、也是对于团队的最基本要求，将它变成基本的制度。
在这之上，保持代码的简洁，再逐步拓展代码的扩展性、复用性，这样得到的代码，对于团队而言，就不再会存在无法接手、阅读困难、改不动这些问题，而对个人而言，在长期的保持注意这些问题的坚持下，也会得到更好的代码质量。
